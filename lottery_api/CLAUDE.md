## 🔬 P3 Shuffle Permutation Test — 對抗驗證 (2026-02-18)

> **核心結論：兩個彩種策略均通過 P3 對抗驗證，確認存在時序訊號。**
>
> 協定：200 shuffles × 1500 periods × seed=42
> 方法：隨機打亂開獎順序，保留號碼分布，破壞時序結構。若 edge 消失 → 有時序訊號。

### P3 驗證結果

| 策略 | Real Edge | Shuffle Mean | p-value | Cohen's d | 判定 |
|------|-----------|-------------|---------|-----------|------|
| **BL 5-bet TS3+M4+FO** | **+1.97%** | +0.22% | **0.030** | **2.13** | **SIGNAL DETECTED** (P3 以舊版 +1.77% 驗證) |
| BL 4-bet TS3+M4 | +1.70% | +0.14% | 0.055 | 1.75 | MARGINAL (Sum-Constraint v2 更新) |
| **PL PP3 3-bet** | **+2.17%** | +0.31% | **0.015** | **2.18** | **SIGNAL DETECTED** |

### 關鍵發現

1. **威力彩 PP3 通過 Bonferroni 修正** (p=0.015 < α/2=0.025)
2. **大樂透 5-bet 通過 5% 門檻** (p=0.030 < 0.05)，但 Bonferroni 邊界 (0.030 > 0.025)
3. **時序訊號佔比**：大樂透 88%，威力彩 86% — Edge 主要來自時序結構，非分布性質
4. **Cohen's d > 2.0** — 兩者均為極大效果量
5. **0/54 Auto-Discovery 方法通過 Bonferroni** — 現有策略已是最佳，無新方法可挖掘

3. **Baseline 必須同注數對齊** - N 注回測必須用 `1-(1-p)^N` 作基準 (2026-01-28 修正)

5. **特別號 V3 是系統亮點** - 1000 期驗證 +2.2% 真實優勢
6. **1000 期 + 多種子是最低驗證標準** - 500 期仍可能是幸運窗口 (2026-02-06 教訓)

   if history[0]['date'] > history[-1]['date']:

   # ✅ 正確

## 🎲 回測驗證規範 (2026-01-28 更新)

> 確保回測結果可復現、可驗證，避免隨機波動造成的誤判。

### 必須遵守的規則

0. **N 注 Baseline 必須對齊** (2026-01-28 新增)：
   ```python
   # ✅ 正確：N 注回測使用 N 注隨機基準
   def correct_baseline(n_bets, p_single):
       """P(N注至少一注 M3+) = 1 - (1 - P(1注))^N"""
       return 1 - (1 - p_single) ** n_bets

   # ❌ 錯誤：用每注個別 M3+ 率作基準
   # 這會低估基準，導致 Edge 虛高
   ```
   > ⚠️ **此規則的違反曾導致 Attention LSTM 和 Zonal Pruning 被錯誤宣布有效**

1. **強制固定隨機種子 (SEED = 42)**：
   ```python
   import numpy as np
   import random

   SEED = 42  # 官方基準種子
   np.random.seed(SEED)
   random.seed(SEED)
   ```

2. **多次運行取統計值**：
   ```python
   # 推薦方案：固定種子作基準 + 多次運行報告置信區間
   results = []
   for seed in range(42, 52):  # 10 次不同種子
       np.random.seed(seed)
       roi = run_backtest()
       results.append(roi)

   print(f"ROI: {np.mean(results):.1f}% ± {np.std(results):.1f}%")
   ```

3. **報告格式規範**：
   - ✅ 正確：`ROI: -65.3% ± 3.2% (N=150, seed=42)`
   - ❌ 錯誤：`ROI: -65.3%`（無法復現）

4. **驗證流程**：
   - 提出者運行 → 記錄種子與結果
   - 驗證者使用相同種子 → 結果應完全一致
   - 若結果不一致 → 腳本有問題，需排查

### 為什麼需要這個規範

| 問題 | 後果 |
|------|------|
| 未固定種子 | 每次運行結果不同，無法驗證 |
| 只跑一次 | 可能恰好「運氣好/差」 |
| 不報告誤差 | 無法判斷結論是否穩健 |

## 🎡 Wheel Coverage System (聰明包牌系統) (2026-02-05 新增)

> **重要哲學轉變**：此系統強調「覆蓋效率」而非「預測」。
> 這是數學保證，不是統計推測。經 Gemini 提出、Claude 驗證同意採納。

### 核心概念

Wheel System 是**組合數學**的應用，不宣稱預測能力：
- 你選一個「候選號碼池」（8-15 個號碼）
- 系統生成最少注數的組合
- **數學保證**：若池中有 M 個號碼被開出，至少一注中 T 個

### 可用覆蓋表

| 池大小 | 保證 | 條件 | 所需注數 | 說明 |
|:---:|:---:|:---:|:---:|:---|
| **8** | 中3 | 若4命中 | **3 注** | ✅ **最經濟** |
| 10 | 中3 | 若3命中 | 10 注 | 寬鬆條件 |
| 10 | 中3 | 若4命中 | 4 注 | 平衡選擇 |
| 12 | 中3 | 若3命中 | 15 注 | - |
| 12 | 中3 | 若4命中 | 8 注 | 常用 |
| 12 | 中4 | 若6命中 | 6 注 | 高保證 |
| 14 | 中3 | 若4命中 | 14 注 | - |
| 15 | 中3 | 若4命中 | 17 注 | - |

### 使用方式

#### CLI Demo（最簡單）
```bash
python3 lottery_api/models/wheel_tables.py
```

#### Python 代碼
```python
from lottery_api.models.wheel_tables import WheelGenerator

generator = WheelGenerator(ticket_size=6)

# 你的候選號碼池
my_pool = [2, 7, 11, 15, 18, 22, 25, 28, 31, 33, 35, 38]

# 生成包牌組合
result = generator.generate(
    pool=my_pool,
    guarantee_t=3,   # 保證中 3 個
    condition_m=4    # 前提是池中有 4 個被開出
)

print(f"需要 {result['ticket_count']} 注")
for i, ticket in enumerate(result['tickets'], 1):
    print(f"第 {i} 注: {ticket}")
```

## 🎓 經驗總結 (Claude + Gemini 聯合)

1. **大樣本才是真相** - 150 期可能出現「幸運窗口」，1500 期才暴露真實
2. **複雜 ≠ 有效** - V11 的 7 專家集成反而比隨機差
3. **Edge vs Random** - 這是唯一有意義的指標
4. **過擬合是大敵** - AI 模型容易「記憶」而非「預測」
5. **簡單策略更穩健** - 偏差互補、Triple Strike 經得起 1500期檢驗 (Cluster Pivot 1500期衰減)
6. **數據防護協議** - 這是系統最重要的資產
7. **~~注數邊際效益遞減~~** - ❌ 已修正 (2026-01-23): Cluster Pivot 6-7 注短期 Edge 高 (+3.5%~+4.4%)，但 **1500期審計為 SHORT_MOMENTUM (-0.45%)**，短期數據不可靠 (2026-02-10)
8. **錨點擴展優於圖分析** - Cluster Pivot > PageRank/度中心性 (2026-01-23 驗證)，但 Cluster Pivot 本身 1500期衰減 (2026-02-10)
9. **使用標準化回測腳本** - 禁止自行編寫回測邏輯，避免數據洩漏 (2026-01-23 新增)
10. **策略與注數要匹配** - V11 7注無效，但 Cluster Pivot 7注有效；策略本身比注數更重要 (2026-01-23 新增)
11. **覆蓋率比複雜度重要** - 威力彩 2注：簡單冷號互補 (Edge +0.45%) 優於複雜 Markov+Stat (Edge -2.05%) (2026-01-25 驗證)
12. **外部建議需獨立驗證** - Gemini 的冷號策略經驗證確實有效，但其他聲稱多數誇大或錯誤 (2026-01-25)
13. **持續優化是唯一出路** - 建立 `strategy_leaderboard.py` 自動辨識最佳窗口 (如 Window 50)，優於固定參數測試。
14. **Baseline 必須與注數對齊** - N 注回測必須用 `1-(1-p)^N` 計算正確隨機基準。此 Bug 曾導致 Attention LSTM (+3.68%→-0.19%) 和 Zonal Pruning (+3.60%→-0.15%) 被錯誤宣布有效。(2026-01-28 修正)
15. **位置排序陷阱** - 彩票號碼按大小排序存儲，ARIMA/Per-Ball LSTM 等「逐位預測」方法本質上只是預測排序統計量的均值，無法捕捉真實模式。(2026-01-28 驗證)
16. **極熱集群攻克** - 針對 115000010 期「32-33-39-43」極熱集群失效分析，開發 `ROI Optimized Ensemble`，通過動能偵測成功將該期命中提升至 M3。(2026-02-04 修正)

17. **誠實系統哲學** - 經 1500 期終極審計，接受彩票是負期望值的隨機遊戲。從「預測幻覺」轉向「覆蓋效率」(Wheel Coverage System)。(2026-02-05 Claude + Gemini 聯合結論)

18. **500 期仍可能是幸運窗口** - 結構過濾(純) 3注在 500 期顯示 Edge +1.92%，但 1000 期崩壞至 -1.38%。**必須 1000 期 + 10 種子穩定性驗證才能採納策略。** (2026-02-06 驗證)

19. **Core-Satellite 覆蓋陷阱** - 理論上錨點共享可提高小獎穩定性，但實測中覆蓋號碼數嚴重不足 (12 vs 18)，在 L1 無真實信號時，覆蓋損失 > 相關收益。大樂透 -0.89%，威力彩 -2.39%。**Layer 2 投注結構無法彌補 Layer 1 預測信號的缺失。** (2026-02-06 驗證)

20. **確定性策略更可信** - 偏差互補 2注在 10 個不同種子下結果完全一致 (方差 0%)，因其演算法無隨機成分。相比之下，含隨機抽樣的策略在不同種子間波動較大。**優先選擇確定性策略。** (2026-02-06 發現)

21. **灰色地帶是結構性盲區** - 115000012期檢討揭示：49% 的命中來自頻率偏差 ±1.5 之間的「灰色地帶」號碼，但現有方法幾乎只從極端偏差取號。**P1 灰色地帶取樣器** 專門解決此問題，1000期驗證 Edge 從 +0.99% 提升至 +1.10%。(2026-02-11 驗證)

22. **隔一期回聲(Lag-2)是穩定信號** - 大樂透 2093 期中 56.9%、威力彩 1884 期中 **67.3%** 的開獎有至少 1 個號碼是 N-2 期的「回聲」。**P0 回聲加分** 補上 Lag-2 信號後：大樂透 1000期 Edge 從 +0.91% 提升至 +1.21%，威力彩 1000期 Edge 從 -0.49% 提升至 +1.11%。(2026-02-11 驗證，跨彩種確認)

23. **P0+P1 跨彩種有效** - 大樂透開發的 P0 回聲加分 + P1 灰色地帶取樣，移植至威力彩 (1-38選6) 後同樣有效：2注 P0 Edge +1.11% (1000期)，3注 P0+P1 Edge +1.01% (1000期)。威力彩的 Lag-2 回聲更強 (67.3% vs 56.9%)，驗證了這是普遍統計現象而非特定彩種的巧合。(2026-02-11 跨彩種驗證)
