### Strategy Summaries

**1. markov_single_biglotto** — 1-bet  
1st-order Markov transition. Uses last draw as seed. Transition matrix built over window=100. Select top-6 by `next_scores`. Deterministic.

**2. markov_2bet_biglotto** — 2-bet (record bet-1 per row)  
Same Markov logic with `num_bets=2`. Top-6 for bet-1, next-6 for bet-2. Replay records bet-1 only, per DAILY_539 convention.

**3. bet2_fourier_expansion_biglotto** — 2-bet (record bet-1 per row)  
FFT period detection over window=500. Score each number by distance-to-dominant-period. Top-12 split into 2 orthogonal bets. Requires `scipy.fft` (already in project venv).

**4. fourier30_markov30_biglotto** — 2-bet (record bet-1 per row)  
Bet-1: Fourier30 (weighted frequency with linear ramp over 30 draws). Bet-2: Markov30 (transition over 30 draws). Enforces max_overlap=3 via `_diversify_biglotto`. Three helper functions must be inlined or imported from `backtest_biglotto_comprehensive.py`.

**5. cold_complement_biglotto** — 2-bet (record bet-1 per row)  
Top-12 coldest numbers from last 100 draws. Bet-1 = coldest 6, bet-2 = next-6. Deterministic. Simplest frequency strategy.

**6. coldpool15_biglotto** — 1-bet  
Same cold-frequency logic as cold_complement but with pool=15 (top-15 coldest). Pick-6 from 15-number cold pool. Trivial pool-size variant of cold_complement.

### Scaffold File

lottery_api/models/p42_wave3_biglotto_adapters.py

Pattern reference: `lottery_api/models/p36_wave2_daily539_adapters.py`

### Pool Constants

```python
_POOL = 49      # BIG_LOTTO number pool 1..49
_PICK = 6       # main numbers per bet
_SPECIAL_POOL = 49  # special number drawn from 1..49
```

### Adapter Input Contract

```python
def predict(history: List[dict]) -> dict:
    """
    history: List of draw records, each with:
        - 'draw'    : str (draw number, e.g. '115000054')
        - 'date'    : str (YYYY/MM/DD, e.g. '2026/05/19')
        - 'numbers' : List[int] — 6 ints in [1,49], sorted ascending
        - 'special' : int — 1 special number in [1,49]

    history MUST be strictly before the target draw (causal slice).
    history MUST be sorted oldest-first.
    """
```

## Special Number Handling

### BIG_LOTTO vs DAILY_539

| Property | BIG_LOTTO | DAILY_539 |
|----------|-----------|-----------|
| Main numbers | 6 from [1,49] | 5 from [1,39] |
| Special number | 1 from [1,49] | None |
| special_hit possible | Yes (if strategy predicts) | Never |

### Wave 3 Strategy Position on Special Numbers

All 6 Wave 3 strategies are **main-number-only** strategies. They do NOT predict the special number.

- `predicted_special = None` for all 6 strategies
- `special_hit = 0` always (because `predicted_special is None`)
- `actual_special` is still recorded from the draw (for future analysis)

This mirrors how DAILY_539 adapters handle the no-special case:
```python
# DAILY_539 pattern (from p36_wave2_daily539_adapters.py):
# "DAILY_539 has no special number — always returns None for special."
```

For BIG_LOTTO Wave 3, the convention extends to:
```python
# BIG_LOTTO Wave 3 — strategies do not predict special
predicted_special = None
special_hit = 0
# actual_special IS recorded from draws.special column
```

### Future Wave 4+ Special Prediction

Wave 4 strategies (e.g. `acb_hot_fourier_3bet_biglotto`) may optionally implement special number prediction. If implemented, they would:
- Set `predicted_special` to an int in [1,49]
- Set `special_hit = 1` if `predicted_special == actual_special`, else 0
- Reference: `lottery_api/models/unified_predictor.py::markov_v2_predict` for inspiration

## Cutoff Semantics

```
prediction_cutoff_date = date of the last draw in history used to generate the prediction
```

**Strict anti-leakage rule**: No draws with `date >= target_date` may appear in `history`.

For 1500-draw rolling window generation:
```python
for i in range(total_draws - 1500, total_draws):
    target = all_draws[i]
    history = all_draws[:i]   # strictly before target
    prediction = adapter.predict(history)
    record = build_record(prediction, target, history[-1])
```

`prediction_cutoff_date` = `history[-1]['date']`  
`history_cutoff_draw` = `history[-1]['draw']`

**Data leakage prevention**: Identical to DAILY_539 implementation. history is always a strict prefix of the sorted draw sequence.
