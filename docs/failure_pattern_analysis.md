# Failure Pattern Analysis Framework
# Phase 4 — Structured Failure Analysis
# 2026-03-15

---

## 目的

將每次預測失敗（M3+ 未達成）轉化為可積累的研究素材，
讓 LLM 從失敗模式中提取結構性改進方向，而非對單期做事後合理化。

---

## 失敗分類體系

### Type A：信號盲區（Signal Blind Spot）
**定義**：開獎號碼完全不在任何策略的候選池中。
**特徵**：ACB rank > 25，MidFreq score < 0.3，無 Markov 支持
**115000065 例子**：#02(rank~24), #05(rank~16) 邊界號，但主注偏中高號

**改進方向**：
- 檢查 ACB boundary_bonus 是否覆蓋不足
- 是否需要 Low-number emergency pool（當 Z1_ZPI > 閾值時啟用）

---

### Type B：信號域分裂（Signal Domain Split）
**定義**：開獎號碼分屬多個不兼容的信號域，無單一注能同時覆蓋。
**特徵**：Fourier top5 ∩ ACB top5 = ∅，且 MidFreq top5 與前兩者也無交集
**常見場景**：Z2=0 後鄰域池盲區、ACB 與 Fourier 信號對立

**改進方向**：
- 信號一致性評分（Consensus Score）：選擇 3+ 信號同意的號碼
- 當 Consensus < 閾值時，改為「覆蓋型」選號而非「集中型」

---

### Type C：Sum 極端尾部（Sum Tail Event）
**定義**：開獎 Sum 在 [μ-2σ, μ+2σ] 之外，所有策略的 Sum 約束失效。
**特徵**：Sum < 60 或 Sum > 140，出現率約 5%
**115000065 例子**：Sum=45（z=-2.41σ），系統性無法預測

**改進方向**：
- Sum 尾部不需要預測，需要「不被排除」（放寬 Sum 過濾條件）
- 監控指標：記錄近 20 期 Sum 均值，若持續偏低則放寬下界

---

### Type D：熱號突然停止（Hot Number Cold Stop）
**定義**：歷史頻率高的號碼突然長期缺席，但策略持續推薦它。
**特徵**：freq_100 > 16%，gap > 20（高頻突斷）
**已確認**：L41 確認「高頻突斷在統計上無預測力」

**改進方向**：當 ACB_freq_deficit 較低但 gap 極長時，降低該號碼權重
（這個矛盾正是 ACB 設計中 freq_deficit 和 gap_score 的張力）

---

### Type E：歷史未見結構（Novel Structure）
**定義**：開獎號碼的組合結構在歷史中出現次數 < 5。
**特徵**：Z1=4 + Z3=0（115000065），或 全奇數（115000018 PL），或連號≥3
**出現率**：各類型約 1-3%

**改進方向**：不強求預測極稀有事件，但設計「不排除機制」（放寬過濾）

## LLM 分析模板

```python
FAILURE_ANALYSIS_PROMPT = """
期號：{period}
開獎號碼：{actual_numbers}
開獎特徵：
  Sum={sum_val} (z={sum_z:.2f})
  Z1={z1}, Z2={z2}, Z3={z3}
  Hot numbers（freq100 >15%）: {hot_numbers}
  Cold numbers（gap >20）: {cold_numbers}

各注預測：
{predictions_table}

各注最佳命中數：{best_match}/5

問題：
1. 這次失敗屬於哪種類型？（A/B/C/D/E，可複選）
2. 哪個具體特徵導致系統無法覆蓋？
3. 若要避免此類失敗，最小改動是什麼？（不允許「重新設計整個策略」）
4. 這個改動有多大概率會損害其他情況的表現？

回答格式：JSON，包含 failure_type, root_cause, minimal_fix, fix_risk_assessment
不得包含：「改變 ACB 基本邏輯」、「增加更多注」、「使用 Zone 信號」（已確認無效）
"""
```

## 批次失敗分析規則

1. **最小期數**：累積 ≥ 20 個失敗期次才做批次分析（避免小樣本過擬合）
2. **類型分布**：計算 A/B/C/D/E 各類型的比例
3. **可操作門檻**：若某類型佔比 < 20%，不針對它設計改動
4. **最高改進優先**：針對佔比最高的失敗類型設計假設，進入 H001-H008 回測隊列
