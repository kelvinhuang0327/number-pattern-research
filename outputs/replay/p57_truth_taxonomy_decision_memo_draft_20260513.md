# P57 — Truth Taxonomy Decision Memo (Draft)

**Date:** 2026-05-13  
**Status:** DRAFT — Pending CTO Decision  
**Author:** Replay Coverage Governance Agent  
**Scope:** Define UI treatment and governance rules per truth level for all 16 registered strategies  
**Non-Scope:** No backfill, no promotion, no mining, no DB writes  

---

## 1. Background

P56 completed a full read-only inventory of all 16 strategies in the Replay Strategy Registry.  
Three truth levels were identified:

| Truth Level | Count | Description |
|---|---|---|
| PRODUCTION_REPLAY | 6 | Actual prediction rows exist in DB from executable adapter |
| DISPLAY_ONLY | 5 | Lifecycle metadata available; no per-draw replay history |
| MISSING_HISTORY | 5 | No artifacts; history not recoverable with current codebase |

This memo proposes governance rules and UI treatment for each truth level.  
CTO must decide which rules to adopt before any coverage extension work begins.

---

## 2. Truth Level Definitions

### 2.1 PRODUCTION_REPLAY
**Definition**: Strategy has ≥ 1 `PREDICTED` row in `strategy_prediction_replays`.  
These rows were generated by an executable `ReplayStrategyAdapter` subclass operating on causal draw history slices (no data leakage).

**Strategies in this class**:
- `power_precision_3bet` — 70 rows / 0 errors
- `power_orthogonal_5bet` — 70 rows / 0 errors
- `biglotto_triple_strike` — 70 rows / 0 errors
- `biglotto_deviation_2bet` — 70 rows / 0 errors
- `daily539_f4cold` — 70 PREDICTED / 20 REPLAY_ERROR ⚠️
- `daily539_markov_cold` — 70 PREDICTED / 20 REPLAY_ERROR ⚠️

**Note on REPLAY_ERROR rows**: The 40 REPLAY_ERROR rows in DAILY_539 strategies do NOT disqualify the strategy from PRODUCTION_REPLAY truth level. They reduce effective coverage for those draws. Root cause investigation is a separate task (P3 scope).

---

### 2.2 DISPLAY_ONLY
**Definition**: Strategy has evaluation evidence (rejection JSON, governance memo, or external artifact) but has no predict tool (`tools/predict_*.py`) and 0 replay rows in DB.

**Strategies in this class**:
- `biglotto_ts3_acb_4bet` — `rejected/ts3_acb_4bet_biglotto.json`
- `biglotto_ts3_markov_freq_5bet` — `rejected/ts3_markov_freq_5bet_biglotto.json` + `strategies/big_lotto/5bet_ts3_markov_freq/`
- `power_shlc_midfreq` — `rejected/shlc_midfreq_power.json`
- `p1_deviation_2bet_539` — `rejected/p1_deviation_2bet_539.json`
- `h6_gate_mk20_ew85` — governance task result not committed to repo

---

### 2.3 MISSING_HISTORY
**Definition**: Strategy has `_LifecycleStub` only. No predict tool, no prediction log, no `rejected/` evidence, no `strategies/` directory. History cannot be recovered without re-implementing the strategy from scratch.

**Strategies in this class** (all RETIRED DAILY_539):
- `acb_1bet`
- `acb_markov_midfreq`
- `acb_markov_midfreq_3bet`
- `midfreq_acb_2bet`
- `midfreq_fourier_2bet`

## 3. Proposed UI Treatment Rules

### 3.1 PRODUCTION_REPLAY → Full Replay Display

| UI Element | Rule |
|---|---|
| Strategy card | Show lifecycle badge (ONLINE) |
| Prediction table | Show per-draw predictions from DB |
| Win/loss result | Show actual vs. predicted per draw |
| Error rows | Display REPLAY_ERROR rows with "Error — prediction unavailable for this draw" |
| Performance summary | Show from `strategies/*/performance_log.json` if present |

**Decision required**: Should REPLAY_ERROR draws be shown as "Error" rows in the UI, or suppressed?  
**Recommendation**: Show with error indicator — suppressing would misrepresent effective coverage.

---

### 3.2 DISPLAY_ONLY → Catalog Metadata Display

| UI Element | Rule |
|---|---|
| Strategy card | Show lifecycle badge (REJECTED / OBSERVATION) |
| Prediction table | **Do not show** — no rows exist |
| Disclaimer | Display: *"This strategy was evaluated but is not in active production. No per-draw prediction history is available."* |
| Evidence link | If `rejected/` JSON exists, may link to evaluation summary |
| Performance summary | Do not show DB-derived performance (0 rows) |

**Decision required**: Should `biglotto_ts3_markov_freq_5bet`'s `strategies/` legacy performance log be surfaced in UI?  
**Recommendation**: No — legacy ID mismatch is unresolved. Surface only the `rejected/` evidence JSON.

**Decision required**: Should `h6_gate_mk20_ew85` be committed to repo so evidence is available in the display?  
**Recommendation**: Yes — commit the task result JSON as a read-only evidence artifact in a future governed docs PR.

---

### 3.3 MISSING_HISTORY → Suppressed or Tombstone

**Option A — Suppress entirely**: Remove from UI catalog. These are early DAILY_539 prototype strategies with no recoverable history and no end-user value.

**Option B — Tombstone**: Show lifecycle badge (RETIRED) with message: *"This strategy has been retired. Historical prediction records are not available."*

**Option C — Registry cleanup**: Remove all 5 RETIRED stubs from `replay_strategy_registry.py`. They remain in git history. UI would no longer reference them.

**Decision required (choose one)**: Option A / Option B / Option C  
**Recommendation**: Option B (Tombstone) — preserves the historical record of the lifecycle without creating false expectation of replay history. Option C requires a separate YES-gated code change.

## 5. Non-Goals (Explicitly Out of Scope)

This memo governs display and classification only. The following are explicitly **NOT** authorized by this memo:

- Backfilling DISPLAY_ONLY or MISSING_HISTORY strategies with retrospective replays
- Promoting any strategy from REJECTED / RETIRED / OBSERVATION to ONLINE
- Writing to `strategy_prediction_replays` DB table
- Running any predict tool against historical draws
- Modifying `replay_strategy_registry.py` (requires separate YES-gated change)
- Removing RETIRED strategies from registry (requires Option C YES gate)
