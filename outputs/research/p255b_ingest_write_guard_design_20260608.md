## Executive Summary

P255B converts the P255A audit findings into a concrete implementable write-guard specification.  The design covers 8 guardrails (G01-G08), a revised BackfillRequest schema, a two-phase dry-run→apply flow with HMAC-based confirm tokens, extended audit logging, pre-write SHA checkpointing, idempotency protection, CORS hardening, and an environment-gated write enable flag. ## G01 — Default dry_run=True

**Current:** `dry_run: bool = False` — Any POST to /api/ingest/backfill that omits dry_run writes to DB

**Proposed:** `dry_run: bool = True`

**When dry_run omitted:** Treated as dry_run=True (safe preview only; no DB write)

**Write path:** Caller must explicitly set dry_run=False AND supply apply_confirmed=True

**Backward compatibility:**
- Breaking for: Any existing caller that relies on dry_run=False being the default
- Safe for: All existing test clients (they pass dry_run=True explicitly)
- Action: Audit all callers and ensure explicit dry_run=False where writes are intended

**Error (dry_run=False without confirm):** HTTP 422 — `write_not_confirmed`

## G02 — Server-Side Confirm Token

The frontend confirmation gate (bfConfirmCheck) is UI-only JavaScript. A direct HTTP POST bypasses it entirely. The server must independently validate that the caller has explicitly opted into a write.

**New BackfillRequest fields:**

| Field | Type | Default | Purpose |
|---|---|---|---|
| `apply_confirmed` | `bool` | `False` | Must be True for dry_run=False to be accepted... |
| `confirm_token` | `Optional[str]` | `None` | Opaque token generated by a prior dry-run response. Encodes:... |
| `requested_by` | `str` | `unknown` | Caller identifier for audit log (e.g. 'ui', 'cli', 'script:p... |
| `reason` | `str` | `` | Human-readable reason for write (e.g. 'fetcher repair backfi... |
| `expected_insert_count` | `Optional[int]` | `None` | Number of draws the caller expects to insert, from prior dry... |

**Server validation order:**
- if dry_run is True: skip all token validation (read-only path)
- if dry_run is False and apply_confirmed is not True: return 422 write_not_confirmed
- if dry_run is False and confirm_token is None: return 422 missing_confirm_token
- if dry_run is False and token is expired (>TTL): return 422 token_expired
- if dry_run is False and token HMAC invalid: return 422 invalid_confirm_token
- if dry_run is False and expected_insert_count provided and actual differs: return 409 count_mismatch
- if all checks pass: proceed with write

**Token format:** `p255-{lottery_type}-{expected_count}-{unix_ts}-{hmac_sha256[:16]}`

**Token TTL:** 300s

**Error responses:**
- Missing token: HTTP 422 `missing_confirm_token`
- Invalid token: HTTP 422 `invalid_confirm_token`
- Count mismatch: HTTP 409 `count_mismatch`

## Request / Response Contract Proposal

**Proposed BackfillRequest schema:**
```
  lottery_type: str = 'BIG_LOTTO'
  draw_list: Optional[List[str]] = None
  dry_run: bool = True  ← CHANGED DEFAULT (G01)
  max_draws: int = 30 (capped 1-500)
  apply_confirmed: bool = False  ← NEW (G02)
  confirm_token: Optional[str] = None  ← NEW (G02)
  requested_by: str = 'unknown'  ← NEW (G04)
  reason: str = ''  ← NEW (G04)
  expected_insert_count: Optional[int] = None  ← NEW (G02/G05)
  idempotency_key: Optional[str] = None  ← NEW (G06)
```

**Validation order:**
- 1. Check INGEST_WRITE_ENABLED (G08) — if false and dry_run=False, return 503
- 2. Check G01: if dry_run=False, require apply_confirmed=True
- 3. Check G02: if dry_run=False, require confirm_token (not None, not expired, HMAC valid)
- 4. Check G06: if idempotency_key present, check for recent duplicate
- 5. Compute pre_sha (G05) and log audit entry start (G04)
- 6. Proceed with BackfillEngine.run()
- 7. Log audit entry complete (G04) with post_sha

**Dry-run response additions (write_guard block):**
- `confirm_token`: str (generated token for apply call)
- `token_expires_at`: ISO-8601 (now + 300s)
- `expected_insert_count`: int (same as inserted count)

**Apply response additions (write_guard block):**
- `db_sha256_before`: str
- `db_sha256_after`: str
- `pre_count`: int
- `post_count`: int
- `idempotency_key`: str | null

**Error response codes:**
- `422_write_not_confirmed`: dry_run=False without apply_confirmed=True
- `422_missing_confirm_token`: dry_run=False without confirm_token
- `422_invalid_confirm_token`: token is malformed or HMAC invalid
- `422_token_expired`: token TTL exceeded (>300s since dry-run)
- `409_count_mismatch`: expected_insert_count != actual pending count
- `409_already_processed`: idempotency_key seen within last 60s
- `409_db_state_changed`: pre_sha does not match expected_db_sha
- `503_write_disabled`: INGEST_WRITE_ENABLED=false

## UI Confirmation Expectations (G03)

**Required flow:**
- 1. User clicks 'Dry-Run Preview' button
- 2. Server returns dry-run summary: total, would-insert count, draw list, confirm_token, expires_at
- 3. UI displays preview table and enables 'Apply Write' button (only if inserted_count > 0)
- 4. User clicks 'Apply Write' → modal dialog appears
- 5. Modal shows: 'You are about to write N draws to the database: [draw list]. This cannot be undone automatically.'
- 6. Modal has two buttons: 'Confirm Write' and 'Cancel'
- 7. On confirm: UI sends POST with dry_run=False, apply_confirmed=True, confirm_token from step 2
- 8. Modal and confirmation state reset after response (success or error)
- 9. Page load / auto-refresh MUST NEVER send POST with dry_run=False

**Forbidden behaviors:**
- Sending dry_run=False on page load
- Sending dry_run=False on periodic auto-refresh
- Reusing a confirm_token across multiple apply attempts
- Leaving the confirmation state persistent across tab reloads
- Enabling 'Apply Write' without first completing a dry-run preview

**Modal copy requirements:**
- Title: "Confirm Database Write"
- Body template: "You are about to write {n} draw(s) to the {lottery_type} database. This action will be logged but cannot be automatically undone."
- Buttons: ['Confirm Write', 'Cancel']

## Audit Log Requirements (G04)

**Log file:** `lottery_api/data/ingest_log.jsonl`

**Required additional fields per write attempt:**
- `dry_run`: bool — was this call dry_run or not
- `trigger_source`: str — 'ui' | 'api_direct' | 'test' | 'script'
- `requested_by`: str — caller identifier from BackfillRequest.requested_by
- `reason`: str — from BackfillRequest.reason
- `pre_count`: int — DB row count for lottery_type before write
- `post_count`: int — DB row count for lottery_type after write (null for dry_run)
- `inserted_draws`: list[str] — draw numbers actually inserted
- `confirm_token_used`: str — first 8 chars of confirm_token (not full for security)
- `expected_insert_count`: int | null — from request
- `actual_insert_count`: int — actual inserts
- `db_sha256_before`: str — SHA256 of DB file before write (null for dry_run)
- `db_sha256_after`: str — SHA256 of DB file after write (null for dry_run or failure)
- `result`: str — 'ok' | 'partial' | 'error' | 'rejected' | 'dry_run'

## Backup / SHA / Integrity Requirements (G05)

**Required steps before any DB insert:**
- 1. Compute SHA256 of lottery_api/data/lottery_v2.db → pre_sha
- 2. Log pre_sha to ingest_log (G04 audit entry)
- 3. Optional: if INGEST_BACKUP_ON_WRITE=true, call sqlite3.connect(db_path).backup(backup_conn)
- 4. Backup file name: lottery_v2_{lottery_type}_{timestamp}.db.bak
- 5. Proceed with db_manager.insert_draws()
- 6. Compute SHA256 post-write → post_sha
- 7. Log post_sha + delta (post_count - pre_count) to ingest_log

**Backup policy:** env `INGEST_BACKUP_ON_WRITE` (default: False), dir `lottery_api/data/backups/`, retention: Keep last 10 backups per lottery_type; auto-prune older ones

## Idempotency / Duplicate Guard (G06)

**layer_1_per_request:** idempotency_key in BackfillRequest — TTL 60s
**layer_2_per_draw:** DB UNIQUE constraint on (lottery_type, draw) — TTL N/As
